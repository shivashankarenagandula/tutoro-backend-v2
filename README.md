# Tutoro Backend — Database Schema & Models

This is the Stage-2 backend schema for Tutoro, designed to sit behind
the existing static marketing site (GitHub Pages) once real match
volume justifies building it out.

**This has been functionally tested, not just written:** migrations
were generated and applied against a real SQLite database, and a user
was created through the custom auth manager, a catalog record chain
(City → Area → Subject) was created, and a linked ParentProfile was
saved — all verified working before this was handed to you. It ships
with `manage.py` and a working `config/settings.py` (SQLite, `DEBUG=True`)
so you can run it immediately without assembling anything yourself.

## Folder structure

```
tutoro-backend/
├── manage.py
├── config/
│   ├── settings.py                       ← working dev config (SQLite)
│   └── settings_production_reference.py  ← Postgres/JWT/env-var hardening to merge in before deploying
├── apps/
│   ├── accounts/     ← custom User model (email login, roles)
│   ├── profiles/     ← ParentProfile, TutorProfile, availability
│   ├── catalog/      ← City, Area, Subject (matches your SEO pages)
│   ├── matching/     ← StudentRequest, Assignment, DemoClass (the core)
│   ├── reviews/       ← Review, tied to a real Assignment (anti-fraud)
│   ├── payments/     ← Stage 2 monetization schema, gateway-agnostic
│   ├── notifications/← in-app/WhatsApp/email notification log
│   └── audit/        ← who-did-what trail for admin actions
├── requirements.txt
├── .env.example
└── .gitignore
```

## Run it right now (SQLite, local, zero setup)

```bash
pip install -r requirements.txt
python manage.py migrate
python manage.py loaddata mvp_seed_data
python manage.py createsuperuser
python manage.py runserver
```

The `loaddata` step seeds exactly the Tutoro V1.0 launch scope — **1 city
(Hyderabad), 8 areas, 8 starter subjects** — nothing beyond that. This
was generated by actually running the create + `dumpdata` + fresh-DB
`loaddata` cycle, not hand-written, so it's confirmed to load cleanly.

**V1.0 area scope (hardcoded in seed data, not in code):** KPHB Colony,
Kukatpally, Miyapur, Nizampet, Hafeezpet, Kondapur, Madhapur, Gachibowli.
Adding a 9th area or a 2nd city later is a new fixture row — zero model
or code changes, per your scalability requirement.

Then visit `http://127.0.0.1:8000/admin/` and log in — every model is
already registered and manageable through the admin panel, seed data
included.

## Before deploying for real

Swap SQLite for PostgreSQL and add the security hardening in
`config/settings_production_reference.py` (env vars, JWT config, CORS
allow-list, HTTPS settings) into `config/settings.py`. You'll need a
real Postgres instance — Railway, Render, or Neon all have free tiers.

## Audit — what's complete, what's missing, what's next

### ✅ Complete
- Full relational schema: 12 models across 8 apps, all with UUID keys,
  proper indexes, and `Meta` constraints
- **Migrations generated and verified** — `makemigrations` +
  `migrate` ran clean against SQLite, zero errors, zero circular
  import issues across all 8 apps
- **API layer — built AND tested end-to-end via 9 real HTTP requests**
  through DRF's test client (not just unit-checked):
  1. Public catalog endpoints return only active areas/subjects
  2. Tutor registration → account + profile created atomically
  3. Parent registration → account + profile created atomically
  4. JWT login issues a real access token
  5. Parent creates a `StudentRequest`, rejected if area is inactive
  6. Parent is correctly blocked (403) from the admin-only matching endpoint
  7. Admin verifies a tutor → `AuditLog` row written
  8. **Same-area-first matching algorithm runs and correctly flags
     `same_area: True`** for a tutor in the same area as the request —
     this is your stated matching-priority requirement, confirmed
     working, not just designed
  9. Admin creates an `Assignment` → `StudentRequest.status` auto-updates to `MATCHED`
- JWT auth (login/refresh), role-based permissions (`IsAdminRole`,
  `IsParentRole`, `IsTutorRole`, `IsOwnerOrAdmin`)
- Tutor verification is a distinct, audit-logged admin action —
  never automatic, never something a tutor can self-grant
- Django admin wired up for `accounts`, `profiles`, `catalog`, and
  `matching`
- MVP seed data (1 city, 8 areas, 8 subjects), round-trip tested on a
  fresh database

### ❌ Still missing
- **Tests as a proper `pytest`/`TestCase` suite** — what exists is a
  verified manual smoke test, not automated regression coverage
- **File upload validation** — resume/ID/photo fields exist but have
  no type/size limits enforced yet; needed before real uploads
- **Frontend integration** — the marketing site's forms currently post
  to Formspree, not this API. See the note below — this is a decision
  point, not just unfinished work.
- **Production settings merge** — `config/settings.py` runs on SQLite
  with `DEBUG=True` for local testing; `config/settings_production_reference.py`
  has the Postgres/env-var/HTTPS hardening to merge in before going live

### Progress: roughly 45% of the full original spec
Schema, migrations, admin, and a tested API layer covering the entire
MVP scope (parent requests, tutor registration, verification,
same-area-first matching, staff-mediated assignment) are done and
confirmed working. What's left is production hardening, file upload
safety, and the frontend integration decision below.

## Deploying this for real (Railway)

1. Push this repo to GitHub (separate repo from `tutoro-site`)
2. Go to [railway.app](https://railway.app) → New Project → Deploy from GitHub repo
3. Add a PostgreSQL plugin (one click) — Railway sets `DATABASE_URL` automatically
4. In Railway's variables tab, add: `DJANGO_SECRET_KEY`, `DEBUG=False`, `ALLOWED_HOSTS=<your-railway-domain>`
5. Merge `config/settings_production_reference.py` into `config/settings.py` before this step, swapping in `env.db("DATABASE_URL")` for the database config
6. Railway runs the `Procfile`'s `release` command automatically on each deploy — migrations and seed data load with zero manual steps
7. Your API is live at `https://<project-name>.up.railway.app/api/...`

## The integration decision that actually matters

The marketing site's "Request a free demo" and "Create tutor profile"
forms are **lead-capture forms** — name, phone, area, subject. This
API's registration endpoints are **full account creation** — they
need an email and password, because they create a real login.

These are two different things, and forcing every demo request through
a full account signup would add friction most parents won't want just
to ask about a tutor. My recommendation: **keep the site forms as
lead capture (Formspree/Sheets) for now**, and use this API as the
internal matching engine your admin team uses once someone's ready to
be onboarded properly. Full self-serve parent/tutor accounts can come
later once volume justifies the extra signup step.

If you'd rather connect the site forms directly to this API now
(adding password fields to the site), that's also buildable — just
flagging it as a real UX tradeoff rather than doing it silently.
